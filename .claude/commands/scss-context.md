## How CSS Vars Are Consumed in Components

Components NEVER hardcode values. They use one of three patterns:

### 1. Direct `var()` reference
```scss
padding: var(--space-base);
font-size: var(--font-size-s);
border-radius: var(--border-radius-soft);
```

### 2. SCSS helper functions (adds semantic override layer)
```scss
// get-background-color($name) → var(--background-color-{name}, var(--color-{name}))
background-color: get-background-color('primary');

// get-border-color($name) → var(--border-color-{name}, var(--color-{name}))
border-color: get-border-color('neutral-3');

// get-text-color($name) → var(--text-color-{name}, var(--color-{name}))
color: get-text-color('neutral-9');
```

### 3. App settings function (theme override layer)
```scss
// get-app-settings-background-color($part, $default) → var(--background-color-{part}, var($default))
background-color: get-app-settings-background-color('header', --color-primary);
```

## Naming Conventions

### Class naming: BEM-hybrid with `osui-` prefix for patterns

- **`src/scss/` classes** (layout, widgets, utilities): no prefix — `.card`, `.btn`, `.alert`, `.layout-*`
- **`src/scripts/` pattern classes**: always `osui-` prefixed — `.osui-accordion`, `.osui-sidebar`, `.osui-balloon`, `.osui-bottom-sheet`
- **Provider override classes**: match the library's own class names — `.vscomp-*`, `.flatpickr-*`

### Class naming: BEM-hybrid
```
.card                  → block
.card-content          → element
.card-sectioned        → modifier/variant
.is-active             → state
.has-accessible-features → state with feature
```

### Widget targeting: data attributes (not classes)
```scss
[data-input], [data-textarea], [data-checkbox], [data-switch], [data-radio]
```

### State classes
```
.is-active  .is-disabled  .is-open  .is-hidden  .is-selected
.is-focused  .is-inline  .is-rtl
.has-accessible-features  .has-event
```

## Pattern-Scoped CSS Custom Properties

Pattern SCSS files frequently declare their own CSS custom properties **scoped to the component root**, then reference global tokens or provide sensible defaults:

```scss
.osui-bottom-sheet {
  --bottom-sheet-max-height: calc(100vh - 54px);
  --border-radius-sharp: none;
  --border-radius-rounded: 16px;          // local override of global --border-radius-rounded
  --osui-bottom-sheet-draggable-area: 56px;
  --osui-bottom-sheet-transition-function: cubic-bezier(0.19, 0.35, 0.56, 0.96);
}

.osui-balloon {
  --osui-floating-position-x: 0;
  --osui-floating-position-y: 0;
  --osui-balloon-shadow: var(--shadow-s);  // aliases a global token
  --osui-balloon-width: max-content;
  --osui-balloon-position: fixed;
}
```

Pattern for naming component-scoped vars: `--osui-{pattern-name}-{property}`.

## Provider Override Pattern

Provider files override third-party library classes but use the same OSUI helper functions and vars throughout:

```scss
// _virtualselect.scss — overrides VirtualSelect library
.vscomp-toggle-button {
  background-color: get-background-color('neutral-0');
  border: var(--border-size-s) solid get-border-color('neutral-5');
  color: get-text-color('neutral-9');
  border-radius: var(--border-radius-soft);
}
```

Pattern files that use providers **import the provider SCSS directly**:
```scss
// _dropdown.scss
@import '../../../../../Providers/OSUI/Dropdown/VirtualSelect/scss/_virtualselect.scss';
```

The `_*_lib.scss` files are the unmodified vendor CSS baselines and should **not be touched** during migration.

## Migration Strategy

**Phase 1 — Dual-var bridge** (non-breaking)
Add a bridge in `_root.scss` that maps OSUI vars to token vars:
```scss
:root {
  --space-base: var(--token-scale-400, 16px);
  --color-primary: var(--token-semantics-primary-base, #1068eb);
  // ...
}
```
Components keep using `--space-base` etc. — no component changes needed.

**Phase 2 — Direct token adoption**
Update component files to use `--token-*` vars directly and remove the bridge.

**Phase 3 — Replace helper functions**
Replace `get-background-color()` / `get-text-color()` etc. with direct `--token-bg-*` / `--token-text-*` references.

**Start with:** `_root.scss` + `_setup-*-vars.scss` files in `00-abstract/`.

**Pattern files** (`src/scripts/OSFramework/OSUI/Pattern/*/scss/`): migrate after the bridge is in place — they should automatically benefit from Phase 1 since they already use `var(--space-*)` etc. Direct token adoption (Phase 2) replaces those with `var(--token-*)`.

**Provider files** (`src/scripts/Providers/OSUI/*/scss/`): migrate the override files (`_virtualselect.scss`, `_flatpickr.scss`, `_noUiSlider.scss`) — leave `_*_lib.scss` vendor files untouched.

**Skip:** `08-servicestudio-preview/`, `10-deprecated/` (unless actively migrating deprecated patterns), `_*_ss_preview_imgs.scss` files (Service Studio only).

## Key Architectural Constraints

1. **AMD output** — no ES module syntax in generated JS; CSS compiles to single bundle per platform.
2. **Dual platform** — O11 and ODC builds differ only in icon library import.
3. **Service Studio preview** — `-servicestudio-*` prefixed properties are real and must not be removed.
4. **Mobile safe areas** — `--os-safe-area-*` pattern wraps both `env()` and CSS var; keep this indirection.
5. **RTL** — `.is-rtl` context selectors are used throughout; logical properties should be preferred in new code.
6. **Data-attribute widgets** — `[data-input]` etc. are the primary styling hooks for form controls; do not convert to classes.
